Q1: FoundationDB separates its architecture this way to keep responsibilities clean and scalable. The Transaction Management System handles the coordination logic, like managing transactions, detecting conflicts, and deciding whether a transaction should commit or abort. The Storage System focuses on storing and serving data, such as handing reads and persisting writes. By separating them, FoundationDB can scale the layers separately and keep the system more flexible and efficient.
Q2: The component responsible for detecting read-write conflicts between concurrent transactions is the resolver. This check happens during the commit process, after the transaction has been assigned a commit version but before the transaction is finalized. 
Q3: The read version represents a snapshot of the database at a specific point in time, When a transaction starts, it is assigned this version and all reads are based on it. This ensures strong consistency, because the transaction sees a stable view of the data.
Q4: Snapshot reads differ from standard reads because they do not add read conflict ranges. This means that they will not cause a transaction to abort if the data changes later. Standard reads track what data was read and will trigger a conflict is that data is modified by a different transaction before commit. Snapshot reads are useful when you want better performance or can tolerate stale data, while standard reads are better when strict correctness is required.
Q5: FDB’s recovery is “cheap” because it doesn’t need to replay logs to rebuild state during recovery. During normal operations, LogServers store committed mutations in order, and StorageServers continuously pull and apply these updates to their local state. For FDB, since StorageServers are mostly up to date, recovery mainly involves reconnecting components and continuing from the latest known version rather than having to pull everything. This design avoids heavy recovery work and allows the system to come back online quickly, making it “cheap”.
